# Production Deployment Roadmap

## Scaling the Crawler

The current implementation uses Python threads and an in-memory queue, which works well for single-machine crawling but has hard limits at production scale. To deploy this as a high-throughput crawler, the first priority is replacing the in-process `queue.Queue` with a distributed message queue such as Redis (using a sorted set keyed by depth) or Apache Kafka. This decouples the crawl frontier from the worker pool, allowing crawler workers to run as independent processes or containers that can be horizontally scaled across machines. Each worker becomes a stateless consumer: dequeue a URL, fetch, parse, publish child URLs back to the queue, write the page record to a shared store. The in-memory `CrawlIndex` should be replaced with a persistent store — Elasticsearch for full-text search capability, or PostgreSQL with a `tsvector` column for a lightweight alternative. The visited set, currently a Python `set` behind a lock, must become a distributed bloom filter (e.g. Redis `BF.ADD`) to support billions of URLs without memory exhaustion. Finally, the single `threading.Lock` bottleneck disappears entirely in a distributed design since each worker process owns no shared state.

## Hardening for Production

Beyond scaling, several reliability and operational concerns must be addressed before this system handles real traffic. The politeness layer needs to evolve from a simple `time.sleep` into a per-domain rate limiter that respects `robots.txt` and `Crawl-Delay` directives — violation of these at scale risks IP bans and legal exposure. The Flask SSE server should be replaced with an async framework (FastAPI with Server-Sent Events or a WebSocket layer) to handle many simultaneous dashboard clients without a thread-per-connection model. Crawl state should be checkpointed to durable storage at regular intervals so a crashed worker or restarted container resumes from where it left off rather than restarting the entire crawl — the bonus persistence requirement in this project is the prototype for this. Observability is critical at scale: structured logging (JSON to stdout), metrics exported to Prometheus, and distributed tracing across worker processes will be necessary to diagnose crawl failures, detect bot-blocking, and measure throughput. Finally, the relevancy model should be upgraded from keyword frequency to TF-IDF scoring at minimum, with BM25 or a lightweight embedding-based reranker as a longer-term investment for search quality.
